## 1. 简介

### 什么是评测系统？

OpenJiuwen 评测系统是一个内置于平台的测试与度量工具，用于衡量你的 AI Workflow 和 Agent 的表现。可以将其理解为一个**质量管控层**，与你的日常运行并行运作。

当你构建了一个 Workflow 或 Agent——例如一个将客户消息路由到正确部门的系统，或一个生成结构化报告的研究 Agent——你自然希望知道：*它真的能正常工作吗？它能稳定运行吗？我修改之后它还能用吗？*

评测系统正是为了回答这些问题而设计的。它能让你：

- **定义测试用例**，描述你的 Workflow 或 Agent 应当处理的输入，以及正确响应的样子
- **自动运行这些测试用例**，针对你空间中的任意 Workflow 或 Agent
- **衡量可靠性**，通过多次运行每个测试来观察成功率
- **阅读详细报告**，精确呈现哪些地方正确、哪些地方出错，以及原因所在

评测系统对 Workflow 和 Agent 的处理方式**完全相同**。本指南中的所有内容对二者均适用，除非有明确说明。

### 为何将其加入 OpenJiuwen？

AI 系统——Workflow、Agent、语言模型链——与传统软件的行为方式不同。传统程序要么正确执行，要么崩溃报错。而 AI Workflow 或 Agent 可能 80% 的时候给出正确答案，20% 的时候给出错误答案；也可能对某类输入完美处理，对另一类输入却彻底失败。

这使得 AI 系统的测试面临独特挑战。评测系统正是为应对这一挑战而构建的，其方式包括：

- **将可靠性视为一个数字**，而非是/否问题
- **多次运行测试**，以便观察真实的通过率
- **不仅检查最终输出**，还检查内部行为（调用了哪些工具、走了哪个分支、Workflow 或 Agent 的结构如何）
- **提供模式感知检查**，能够理解 OpenJiuwen Workflow 和 Agent 可能呈现的结构模式

### 它解决了哪些问题？

| 问题 | 没有评测时 | 有了评测后 |
|------|-----------|-----------|
| "我的改动破坏了什么吗？" | 手动测试，祈祷没问题 | 运行评测套件，立即得知 |
| "这个 Workflow 或 Agent 有多可靠？" | 凭感觉，不确定 | 以精确百分比衡量的通过率 |
| "它的路由是否正确？" | 难以系统性验证 | 模式感知检查可检测路由行为 |
| "哪个版本的 Workflow 或 Agent 更好？" | 基于几次尝试猜测 | 并排比较运行结果 |
| "它准备好上生产了吗？" | 凭直觉 | 可量化的质量门禁 |

## 2. 核心概念

### 2.1 评测任务

一个**评测任务**是一个单独的测试用例，包含：
- **名称**
- **输入数据**——发送给 Workflow 或 Agent 的内容
- **预期输出**——正确响应的样子
- **评分器**——用于判断运行是否通过的规则
- **试运行次数**——该任务独立执行的次数
- **难度**标签（简单 / 中等 / 困难）——仅用于显示，不影响执行
- **模式类型**标签——设置后会验证执行结构；若只需检查输出，可留空

### 2.2 评测套件

一个**评测套件**是相关任务的集合。你可以一次性运行整个套件，针对特定的 Workflow 或 Agent，并获得一份整合报告。

### 2.3 试运行

一次**试运行**是对任务的单次执行。如果一个任务设置了 `trials: 5`，系统会对该任务独立运行五次。每次试运行完全独立——全新执行，赋予新的会话 ID。

### 2.4 评分器

**评分器**是判断一次试运行通过还是失败的逻辑单元。

#### 确定性评分器
应用固定规则，不涉及 AI。类型包括：`output_check`、`state_check`、`tool_call_check`、`transcript_check`、`pattern_check`。即时执行，无成本，结果可复现。

#### 基于模型的评分器
使用大模型根据评分标准评估输出质量。对于细致、主观的评判标准更灵活。速度较慢且消耗 Token。

#### 基于代码的评分器
自定义 Python 函数，完全掌控评分逻辑。

### 2.5 模式感知评测

| 模式 | 含义 |
|------|------|
| **Routing** | 发生了条件判断（使用了 IF 组件） |
| **Chaining** | 两个或更多步骤顺序执行 |
| **Parallelisation** | 多个分支同时执行 |
| **Orchestrator–Worker** | 顶层 Workflow 将任务委托给子 Workflow |
| **Evaluator–Optimizer** | 执行在循环中运行以进行优化改进 |
| **Memory Usage** | Workflow 读取或写入了持久化状态 |

### 2.6 pass@k

**pass@k** = k 次独立抽样中至少 1 次通过的概率。
- pass@1 = 普通通过率
- pass@3 = 90% 表示：尝试 3 次，有 90% 的概率至少成功一次

### 2.7 pass^k

**pass^k** = k 次独立抽样全部通过的概率。
- pass^3 = 22% 表示三次运行全部通过的概率仅为三分之一左右

### 2.8 执行追踪

一次试运行中发生的所有事情的详细日志：激活的组件、调用的工具、走过的分支、延迟、Token 用量、错误信息。

### 2.9 基准测试 vs 回归测试

**基准测试套件**是以绝对方式衡量能力的标准化测试集合。OpenJiuwen 预置了 7 套基准测试。

**回归测试套件**是你为特定 Workflow 或 Agent 自行构建的测试集合，随时间不断丰富，用于防止功能回退。

## 3. 评测系统的工作原理

### 3.1 OpenJiuwen 正常运行 Workflow 或 Agent 的方式

正常情况下：提供输入 → Workflow 执行 → 收到输出。没有任何机制检查输出是否正确。

### 3.2 "带评测运行"时有何不同

评测系统为每个任务运行 N 次试运行，捕获执行追踪，将其传递给评分器，收集结果，并计算聚合指标。

### 3.3 评测框架如何封装执行过程

评测框架的工作流程：加载任务 → 运行试运行（顺序或并行）→ 收集追踪信息 → 发送给评分器 → 保存结果 → 计算指标 → 将运行状态更新为已完成。

### 3.4 评分器如何分析结果

- 确定性评分器：立即应用规则，记录预期值/实际值/条件
- 基于模型的评分器：将输出和评分标准发送给大模型，接收评判结果
- 基于代码的评分器：执行自定义 Python 函数

多个评分器时：只有每个权重不为零的评分器全部通过，该次试运行才通过。得分 = 加权平均值。

### 3.5 指标的计算方式

所有试运行完成后，系统计算以下指标：

**通过 / 失败**
- **成功率** — 通过的试运行占总数的比例
- **错误率** — 发生执行错误的试运行比例

**采样指标**
- **pass@k** — k 次抽样中至少 1 次通过的概率
- **pass^k** — k 次抽样全部通过的概率

**得分质量**（当评分器返回数值得分时）
- **平均得分** — 所有试运行的平均评分器得分
- **得分标准差** — 标准差
- **满分率** — 得分恰好为 1.0 的试运行比例
- **得分分布** — 直方图：每个 20% 得分区间内的试运行占比

**延迟**
- **平均延迟**、**中位延迟（p50）**、**p95 延迟**、**最小 / 最大延迟**

**Token 用量**
- **总 Token 数**、**每次试运行 Token 数**

**可靠性**（仅限多次试运行的任务）
- **波动性（Flakiness）** — 各唯一任务通过/失败标准差的均值

**按评分器分项统计**
- 按评分器名称分组的通过率、平均得分和试运行计数

**自定义指标**（如果套件中有定义）
- 用户编写的 Python `compute(results)` 函数

### 3.6 结果的存储与展示方式

每次试运行结果都存储在数据库中。结果与特定的 Workflow 或 Agent 版本绑定。Workflow 或 Agent 名称在运行开始时捕获。

## 4. 在前端使用评测功能

### 4.1 导航到评测页面

1. 在左侧边栏中点击**评测**
2. 左侧面板：评测套件列表；右侧面板：详细视图，包含三个标签页：**任务**、**运行记录**、**结果与追踪**

### 4.2 创建评测套件

**方案 A — 创建空套件：**
1. 点击 **+ 新建套件**
2. 输入套件名称和可选说明
3. 点击**创建**

**方案 B — 导入预置基准测试：**
1. 点击**加载基准测试**
2. 从 7 套预置基准测试中选择
3. 可选输入自定义名称
4. 点击**导入**

### 4.3 向套件添加任务

1. 选择套件 → 点击**任务**标签页 → 点击 **+ 添加任务**
2. 填写：Task ID、Task Name、Description、Pattern Type、Difficulty、Trials、Input Data、Expected Output、Graders
3. 点击**添加任务**

### 4.4 运行评测

1. 点击 **▶ 运行评测**
2. 选择目标类型（Workflow 或 Agent），选择目标，可选开启并行执行
3. 点击**开始运行**

### 4.5 查看运行记录

| 列名 | 描述 |
|------|------|
| **Run ID** | 缩短的唯一标识符 |
| **目标** | 被评测的 Workflow 或 Agent 名称 |
| **状态** | 待处理 → 运行中 → 已完成 / 失败 / 已取消 |
| **成功率** | 通过的试运行百分比 |
| **查看结果** | 打开完整结果视图的按钮 |
| **删除** | 垃圾桶图标——删除该运行记录 |

**查看结果**按钮显示：
- **"查看结果"** — 针对已完成的运行
- **"实时结果"** — 针对正在进行中的运行

### 4.6 查看结果与追踪

在任意非待处理状态的运行上点击**查看结果**或**实时结果**。结果视图包含四个子标签页：**概览**、**指标**、**评分器**、**追踪**。

### 4.7 四个结果子标签页

#### 概览标签页——统计卡片

**通过 / 失败组**
| 卡片 | 显示内容 |
|------|----------|
| **成功率** | 通过百分比。≥80% = 绿色，50–80% = 琥珀色，<50% = 红色。 |
| **已通过** | 通过的试运行原始计数。 |
| **已失败** | 失败的试运行原始计数。 |
| **错误率** | 仅在发生错误时显示。 |
| **波动性（Flakiness）** | 各任务通过/失败标准差的均值。仅在多次试运行任务中显示。 |

| **总 Token 数** | 无得分数据时在此处显示总 Token 数。 |

| **k** | 独立抽样次数 |

### 4.8 管理自定义聚合指标

1. 选择套件 → 点击标题栏中的 **Σ**（sigma）图标
2. 自定义指标对话框列出现有指标
3. 点击 **+ 新建指标**，输入名称、说明和 Python 代码
4. 点击**保存**

`compute(results)` 函数接收包含以下字段的试运行结果字典：`task_id`、`passed`、`score`、`latency_ms`、`token_usage`、`error_message`、`grader_results`。

示例：
```python
def compute(results):
    if not results:
        return 0.0
    high_quality = sum(
        1 for r in results
        if r.get("passed") and r.get("score", 0) >= 0.8
    )
    return high_quality / len(results)
```

每次查看结果时，自定义指标都会重新计算。

### 4.9 查看每个任务的详细结果（追踪查看器）

**追踪**标签页按任务展示结果。每个任务显示：
- 任务名称及通过的试运行数量（例如"2 / 3 通过"）
- 各次试运行结果，以可展开的手风琴行呈现

每个折叠的试运行行显示：通过/失败、N/M 个评分器通过、延迟、Token 用量、Trace ID。

展开试运行后显示：
1. 执行错误（如有）——红色横幅
2. Workflow 输出——以绿色/红色高亮
3. **评分器详情表格** — 每个评分器一行：✓/✗、名称、得分、预期值/实际值/条件
4. Token 用量分项

## 5. 理解评测结果

### 5.1 "通过"和"失败"的含义

当每个评分器都通过时，一次试运行才算通过。试运行失败并不总意味着 Workflow 存在问题——可能是评分器标准过于严格，或预期输出未涵盖所有可接受的响应形式。

### 5.2 如何解读成功率

| 成功率 | 解读 |
|--------|------|
| **90–100%** | 优秀。高度可靠。 |
| **70–90%** | 良好。或许值得小幅改进。 |
| **50–70%** | 一般。通过次数多于失败次数。 |
| **30–50%** | 较差。接近一半的时候会失败。 |
| **<30%** | 很差。在这些任务上基本无法正常工作。 |

### 5.3 如何解读 pass@k 和 pass^k

- 如果可以多次尝试 → 关注 **pass@k**。pass@3 = 97% 表示：尝试 3 次，几乎可以确定至少有一次成功。
- 如果每次执行都必须成功 → 关注 **pass^k**。pass^3 = 34% 表示三次全部通过的概率仅约三分之一。

### 5.4 如何解读得分质量指标

**平均得分** — 所有试运行的平均得分。高得分但失败（near-miss）vs. 低得分失败（complete failure）传达不同信息。

**满分率** — 得分为 1.0 的试运行比例。高成功率 + 高满分率 = 表现出色。

**一致性（得分标准差）**：
- 高（标准差 ≤ 5%）——每次质量非常相近
- 中（标准差 5–15%）——存在一定变化
- 低（标准差 > 15%）——质量波动较大

**波动性（Flakiness）**：
- 0.000 = 完全稳定
- 0.1–0.2 = 轻度波动
- >0.3 = 高度波动；具有明显的不确定性

**得分分布（· 详情弹窗）** — 5 个区间的直方图，展示得分的分布情况。

### 5.5 如何读取评分器输出

评分器详情表格显示：✓/✗、得分（0–100%）、预期值、实际值、条件。

- `actual: null` = 无输出或字段未找到
- 基于模型的评分器失败：查看 `feedback` 字段
- 基于代码的评分器失败：查看 `error` 字段

### 5.6 如何理解模式检查

| 模式 | 常见失败原因 |
|------|-------------|
| Routing | 未使用 IF 组件，或走了错误的分支 |
| Chaining | 只运行了一个步骤 |
| Parallelisation | 组件顺序运行而非并行运行 |
| Orchestrator–Worker | 未调用子 Workflow |
| Evaluator–Optimizer | 循环未执行 |
| Memory Usage | 变量未被读取或写入 |

### 5.7 如何读取执行追踪

1. 执行错误横幅（红色）——如果抛出异常
2. 评分器详情表格——每个评分器的预期值与实际值对比
3. Trace ID——用于与服务器日志交叉参考

### 5.8 如何识别功能回退

1. 修改前运行套件 → 记录结果
2. 进行修改
3. 再次运行套件
4. 比较成功率

### 5.9 如何比较多次运行

运行记录标签页按时间顺序列出所有运行。打开运行 A 的结果，记录失败项。打开运行 B，记录失败项。差异即显示你的修改修复了什么或破坏了什么。

## 6. 创建和管理评测套件

### 6.1 什么是评测套件

一个有命名的相关任务容器，隶属于某个空间。

### 6.2 套件的组织方式

每个套件包含：唯一 ID、名称、说明、任务列表、运行历史记录。同一套件可以针对不同的 Workflow 或 Agent 运行。

### 6.3 基准测试套件 vs 回归测试套件

**预置基准测试套件**：客观、标准化，测试抽象模式。

**自定义回归测试套件**：针对特定 Workflow 或 Agent，随时间不断丰富，防止功能回退。

### 6.4 任务的定义方式

1. 发送什么输入？
2. 正确响应是什么样的？
3. 选择哪种评分器？（确定性 / 基于模型 / 基于代码）
4. 需要几次试运行？（1=冒烟测试，3=良好平衡，5-10=准确的可靠性数据）
5. 结构是否重要？（设置 Pattern Type 或留空）
6. 难度如何？（设置 Difficulty 用于分类整理）

### 6.5 评分器配置示例

**检查特定字段：**
```json
[{"name": "result_equals_6", "grader_type": 0, "config": {"check_type": "state_check", "path": "result", "expected_value": 6.0, "condition": "eq"}}]
```

**大模型质量检查：**
```json
[{"name": "response_quality", "grader_type": 1, "config": {"model_id": "your-model-id", "rubric": "The response is helpful and provides a clear next step.", "passing_score": 0.7}}]
```

### 6.6 如何选择运行哪个套件

- "这个东西能正常工作吗？" → 基础冒烟测试套件
- "我的改动破坏了什么吗？" → 回归测试套件
- "它准备好上生产了吗？" → 回归测试套件 + 基准测试套件

## 7. 何时以及为何使用评测

### 7.1 测试新的 Workflow 和 Agent
部署前先快速评测。即使只有 5–10 个测试用例，也能发现明显的问题。

### 7.2 比较 Workflow 版本
针对 v1 和 v2 运行相同的套件，客观比较成功率。

### 7.3 比较 Agent 版本
针对 Agent v1 和 Agent v2 运行相同套件，通过比较判断某次改动是否提升了性能。

### 7.4 检测功能回退
每次进行非微小改动时，运行回归测试套件。

### 7.5 验证路由逻辑
用多个示例测试每个路由类别。模式检查验证 IF 组件是否被使用。

### 7.6 确保并行分支行为正确
并行化模式检查验证分支是否真的并行运行，而非顺序运行。

### 7.7 检查 Orchestrator–Worker 分解
模式检查确认子 Workflow 是否被调用。工具调用检查验证子任务的输入。

### 7.8 测试确定性函数
对精确值验证使用 `state_check` 配合 `condition: "eq"`。

### 7.9 持续提升可靠性
为每个生产问题添加测试用例。每次改动后运行套件。逐步构建全面的安全防护网。

## 8. 术语表

| 术语 | 定义 |
|------|------|
| **Agent** | OpenJiuwen 中具有 AI 能力的实体，能够执行操作、使用工具并维持对话。 |
| **Avg Score** | 所有试运行的平均评分器得分（0.0–1.0）。 |
| **Benchmark** | 以绝对方式衡量能力的标准化测试用例集合。 |
| **Code-Based Grader** | 运行自定义 Python 函数的评分器。 |
| **Consistency** | 得分标准差的反面——"高"表示每次得分相近。 |
| **Custom Aggregate Metric** | 用户自定义的 `def compute(results)` 函数，存储在套件 `config.custom_metrics` 中。 |
| **Deterministic Grader** | 应用固定规则、不调用大模型的评分器。 |
| **Difficulty** | 分类标签（简单/中等/困难）。不影响执行。 |
| **Error Rate** | 发生执行异常的试运行比例。 |
| **Evaluation Harness** | 内部组件，负责编排试运行执行、评分和指标计算。 |
| **Evaluation Run** | 针对某个 Workflow 或 Agent 完整执行一次套件的过程。 |
| **Evaluation Suite** | 相关评测任务的集合。 |
| **Execution Trace** | 一次试运行的详细日志：组件、工具、分支、延迟、Token。 |
| **Flakiness** | 各唯一任务在多次试运行中通过/失败标准差的均值。0 = 稳定，0.5 = 随机。 |
| **Grader** | 评估一次试运行输出的组件：判断通过/失败 + 质量得分。 |
| **Graders Tab** | 结果子标签页，显示各评分器的通过率、平均得分和试运行计数。 |
| **Metrics Tab** | 结果子标签页，显示 pass@k/pass^k 表格和自定义指标。 |
| **Model-Based Grader** | 使用大模型根据评分标准评估输出的评分器。 |
| **Overview Tab** | 结果子标签页，显示高级统计卡片。 |
| **p95 Latency** | 第 95 百分位延迟——95% 的试运行完成时间低于此值。 |
| **pass@k** | k 次独立试运行中至少 1 次通过的概率。 |
| **pass^k** | k 次独立试运行全部通过的概率。 |
| **Pattern Type** | 六种结构模式之一：Routing、Chaining、Parallelisation、Orchestrator–Worker、Evaluator–Optimizer、Memory Usage。 |
| **Pattern Check** | 验证执行追踪是否呈现预期结构模式的评测。 |
| **Perfect Score Rate** | 得分为 1.0 的试运行比例。 |
| **Per-Grader Breakdown** | 按评分器名称汇总的统计数据：通过率、平均得分、计数。 |
| **Regression** | 原本正常工作的功能在改动后停止工作。 |
| **Regression Test Suite** | 针对特定 Workflow 或 Agent 的测试用例集合，用于检测功能回退。 |
| **Rubric** | 用于基于模型的评分器的自然语言输出质量描述。 |
| **Score** | 表示试运行质量的数字（0.0–1.0），由评分器给出。 |
| **Score Distribution** | 五个 20% 区间内得分分布的直方图。通过平均得分卡片上的"· 详情"访问。 |
| **Space** | OpenJiuwen 中的组织单元。 |
| **state_check** | 通过点路径读取字段并与预期值比较的确定性评分器。 |
| **Success Rate** | 所有试运行中通过的比例。 |
| **Task** | 单个测试用例：输入、预期输出、试运行次数、评分器。 |
| **Tokens Per Trial** | 每次单独试运行的平均 Token 数。 |
| **Traces Tab** | 结果子标签页，显示每次试运行的可展开详细信息。 |
| **Trial** | 一次任务的独立执行。 |
| **Weighted Aggregation** | 组合评分器得分的方式：Σ(score_i × weight_i) / Σ(weight_i)。 |
| **Workflow** | OpenJiuwen 中处理输入并生成输出的结构化组件图。 |

## 9. 常见问题

**问：使用评测系统需要编写代码吗？**
不需要。创建套件、加载基准测试、添加任务、运行评测和查看结果，这些操作均可通过界面完成。

**问：可以用同一个套件评测 Workflow 和 Agent 吗？**
可以。点击"运行评测"时选择"Workflow"或"Agent"即可。

**问：应该用几次试运行？**
3 次试运行是一个好的起点。重要测试用例可用 5 次。基础冒烟测试用 1 次即可。

**问：成功率 70%，这算好还是不好？**
取决于使用场景。创意写作达到 70% 是可以接受的。财务计算达到 70% 则不可接受。

**问：pass@k 和成功率有什么区别？**
成功率 = 通过的原始比例。pass@1 = 相同。pass@k（k>1）= k 次运行中至少一次成功的概率。pass@k 始终高于成功率。

**问：评测可以在后台运行吗？**
可以。点击"开始运行"后，离开页面，稍后返回即可查看结果。

**问：如果试运行因错误崩溃会怎样？**
该次试运行被记录为失败，完整错误信息显示在追踪查看器中。评测继续处理剩余的试运行。

**问：成功率和平均得分有什么区别？**
成功率是二元的（通过/失败的比例）。平均得分是连续的——即使失败的试运行也有得分。成功率 50% 且平均得分 0.75 = 接近通过的失败；平均得分 0.20 = 完全失败。

**问："波动性（Flakiness）"能告诉我成功率无法反映的什么信息？**
成功率 = 总体通过频率。波动性 = 失败是随机的还是特定于某种输入的。零波动性 = 相同输入总是失败，是确定性 Bug。高波动性 = 随机失败，是不确定性问题。

**问：评分器标签页有什么用？**
显示各评分器的通过率和平均得分。当你配置了多个评分器并想知道哪个具体判断条件导致失败时使用。

**问：什么是自定义聚合指标？**
从所有试运行结果计算套件级别指标的 Python 函数。通过 **Σ** 按钮定义，每次查看结果时重新计算。

**问：为什么某些统计卡片上有"· 详情"链接？**
它们会打开弹出对话框：
- **平均得分 · 详情** → 得分分布直方图
- **平均延迟 · 详情** → 延迟分项表格（中位数、p95、最小值、最大值、总计）

**问：延迟是如何测量的？**
从试运行开始到结束的挂钟时间，单位为毫秒。包括大模型时间、工具调用时间和所有处理时间。不包括评分器执行时间。

**问：评测结果仅限于我的空间可见吗？**
是的。所有评测数据均限定在你的空间范围内。
