# OpenJiuwen 评测系统

OpenJiuwen Workflow 与 Agent 的模式感知基准测试与评测系统。

## 概述

评测系统允许你定义声明式测试套件，针对 Workflow 或 Agent 运行这些测试，并在一套全面的指标体系中衡量正确性、模式合规性及性能表现。

```
Evaluation Suite → Tasks → Trials → Graders → Metrics
```

## 快速开始

### 1. 创建评测套件

```bash
POST /api/v1/evaluation/create
{
  "suite_name": "My Routing Tests",
  "description": "Tests for conditional routing",
  "space_id": "your-space-id"
}
```

### 2. 添加任务

```bash
POST /api/v1/evaluation/task/add
{
  "evaluation_id": "<来自第 1 步>",
  "task_name": "Positive sentiment routing",
  "task_definition": "...",
  "input_data": {"message": "I love this!"},
  "expected_output": {"branch": "positive"},
  "graders_config": [{...}],
  "trials": 3,
  "pattern_type": 0,
  "space_id": "your-space-id"
}
```

### 3. 运行评测

```bash
POST /api/v1/evaluation/run/start
{
  "evaluation_id": "<套件 ID>",
  "workflow_id": "your-workflow-id",
  "space_id": "your-space-id"
}
# 立即返回 run_id；评测在后台运行
```

### 4. 查看结果

```bash
GET /api/v1/evaluation/results/{run_id}
```

## 支持的 Workflow 模式

| 模式 | 值 | 检测方式 |
|------|----|----------|
| ROUTING | 0 | 执行追踪中存在 IF 组件 |
| CHAINING | 1 | 存在 ≥2 个顺序执行的组件跨度 |
| PARALLELIZATION | 2 | 执行时间窗口存在重叠 |
| ORCHESTRATOR_WORKER | 3 | 使用了 SUB_WORKFLOW 组件 |
| EVALUATOR_OPTIMIZER | 4 | 使用了 LOOP 组件 |
| MEMORY_USAGE | 5 | 使用了 SET_VARIABLE 或 VARIABLE_MERGE |

## 指标

所有指标均在 `metrics.py` 中计算，并在运行结果的 `metrics` 键下返回。

### 通过 / 失败

| 指标 | 描述 |
|------|------|
| `success_rate` | 通过的试运行占总试运行的比例（0.0–1.0） |
| `passed` | 通过的试运行原始计数 |
| `total_results` | 已执行的试运行总数 |
| `error_rate` | 发生执行错误的试运行比例 |
| `total_tasks` | 参与评测的唯一任务数量 |
| `task_pass_rate` | 至少有 1 次试运行通过的唯一任务占比 |
| `tasks_fully_passed_rate` | 全部试运行均通过的唯一任务占比 |
| `tasks_never_passed_rate` | 没有任何试运行通过的唯一任务占比 |

> 仅当 `total_tasks < total_results`（即多次试运行）时，`total_tasks`、`task_pass_rate`、`tasks_fully_passed_rate` 和 `tasks_never_passed_rate` 才在界面中显示。

### 采样指标

| 指标 | 描述 |
|------|------|
| `pass_at_k` | `{1: p, 3: p, 5: p}` — k 次抽取中至少 1 次通过的概率 |
| `pass_pow_k` | `{1: p, 3: p, 5: p}` — k 次抽取全部通过的概率 |

### 得分质量（当评分器返回得分时）

| 指标 | 描述 |
|------|------|
| `avg_score` | 所有试运行的平均评分器得分 |
| `median_score` | 第 50 百分位评分器得分——对异常值的敏感度低于均值 |
| `score_std` | 标准差——越低表示越稳定 |
| `score_min` / `score_max` | 观测到的得分范围 |
| `perfect_score_rate` | 得分恰好为 1.0 的试运行比例 |
| `score_distribution` | 直方图：每个 20% 区间内的试运行占比（`0_20`、`20_40`、`40_60`、`60_80`、`80_100`） |

### 延迟

| 指标 | 描述 |
|------|------|
| `avg_latency_ms` | 平均试运行执行时间 |
| `median_latency_ms` | 第 50 百分位延迟 |
| `p75_latency_ms` | 第 75 百分位延迟 |
| `p95_latency_ms` | 第 95 百分位延迟——大多数试运行的最差情况 |
| `min_latency_ms` | 最快的试运行 |
| `max_latency_ms` | 最慢的试运行 |
| `total_latency_ms` | 所有试运行延迟之和 |
| `latency_std_ms` | 延迟标准差——执行时间的波动程度 |
| `latency_cv` | 变异系数（标准差 ÷ 均值）——低 = 延迟可预测，高 = 延迟不稳定 |

### Token 用量

| 指标 | 描述 |
|------|------|
| `token_usage` | `{prompt_tokens, completion_tokens, total_tokens}` — 聚合值 |
| `tokens_per_trial` | `{prompt_tokens, completion_tokens, total_tokens}` — 每次试运行平均值 |
| `tokens_efficiency` | 按结果分类的平均 Token 用量：`{"passed": {prompt_tokens, completion_tokens, total_tokens}, "failed": {…}}` |

### 可靠性

| 指标 | 描述 |
|------|------|
| `flakiness` | 各任务在多次试运行中通过/失败标准差的均值。`0.0` = 完全稳定，`0.5` = 随机。当没有任何任务执行多次试运行时为 `null`。 |
| `latency_cv` | 延迟变异系数（可靠性角度）——衡量执行时间是否可预测。 |
| `tasks_never_passed_rate` | 从未在任意试运行中成功的任务占比（仅多次试运行）——用于识别持续失败的任务。 |
| `tasks_fully_passed_rate` | 每次试运行均通过的任务占比（仅多次试运行）——衡量重复采样下的鲁棒性。 |

### 按评分器分项统计

| 指标 | 描述 |
|------|------|
| `per_grader_breakdown` | `{grader_name: {pass_rate, avg_score, count}}` — 按评分器聚合 |

### 自定义聚合指标

用户自定义的 Python 函数，存储在套件的 `config.custom_metrics` 列表中。每条定义格式如下：

```json
{
  "name": "my_metric",
  "description": "可选说明",
  "code": "def compute(results):\n    return sum(r['score'] for r in results) / len(results)"
}
```

`compute(results)` 函数接收完整的试运行结果字典列表，必须返回 `float` 或 `dict`。自定义指标显示在结果视图的**指标**标签页中的"Custom Metrics"区域。

## 结果界面——标签页结构

结果视图（`EvaluationResults.tsx`）包含四个子标签页：

| 标签页 | 内容 | 显示条件 |
|--------|------|----------|
| **概览** | 统计卡片：成功率、通过/失败计数、得分、延迟、Token | 始终显示 |
| **指标** | pass@k / pass^k 表格 + 自定义指标表格 | 有数据时显示 |
| **评分器** | 各评分器通过率、平均得分、试运行计数 | 有评分器数据时显示 |
| **追踪** | 每次试运行可展开的详细信息及评分器结论 | 始终显示 |

## 运行测试

```bash
cd backend
pytest openjiuwen_studio/core/executor/evaluation/tests/ -v
```

测试使用 `conftest.py` 对 `openjiuwen` 核心库进行存根处理，因此无需完整运行时环境即可独立运行。
