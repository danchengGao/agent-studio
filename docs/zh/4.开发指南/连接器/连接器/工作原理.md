# 人类如何通过连接器连接到OpenJiuwen

本文档详细说明每个阶段发生的事情 — 从获取平台凭据到用户在WhatsApp或Slack中运行工作流。适合首次设置连接器的开发者，或任何希望了解完整连接流程的人。

---

## 整体架构

```
  您(开发者)
       │
       │  一次性配置(平台机器人凭据)
       ▼
  连接器服务器启动
       │
       │  连接到消息平台
       ▼
  消息平台  ◄──────────►  连接器机器人  ──────────────►  OpenJiuwen后端
  (Telegram、Slack、    消息        (channels/                HTTP + 每用户
   WhatsApp等)                    platforms/<name>/)         Bearer令牌
       │
       │  人类输入/login，运行智能体，运行工作流
       ▼
  终端用户(人类)
```

连接器的方向是: **人类 → 消息平台 → 连接器机器人 → OpenJiuwen后端**。
人类是消费者。OpenJiuwen是被暴露的系统。
这与MCP相反——在MCP中，LLM作为Claude自主消费OpenJiuwen。

**两组凭据并列存在:**

| 凭据 | 归属 | 设置时机 | 作用 |
|------|------|---------|------|
| 平台机器人令牌 | 机器人本身(Telegram、Slack…) | 开发者，在启动时 | 让机器人连接到消息平台 |
| OpenJiuwen Bearer令牌 | 每个人类用户 | 用户，在运行时通过/login | 该用户以自己的身份调用OpenJiuwen |

## 完整步骤流程

```
┌─────────────────────────────────────────────────────────────────────┐
│ 第1步 — 获取平台凭据  (一次性，由开发者完成)                          │
│                                                                     │
│  每个消息平台需要自己的机器人凭据。                                   │
│  这些凭据从平台的开发者控制台获取一次，                               │
│  与OpenJiuwen无关。                                                  │
│                                                                     │
│  示例:                                                              │
│    Telegram:  通过@BotFather创建机器人 → 获取机器人令牌              │
│    Slack:     创建Slack应用 → 获取Bot User OAuth Token              │
│    WhatsApp:  通过Meta Business注册 → 获取访问令牌                   │
│               + phone_number_id + verify_token                      │
│    Discord:   创建应用 → 获取机器人令牌                              │
│    Teams:     注册Azure机器人 → 获取app_id + app_password           │
│                                                                     │
│  这些凭据向消息平台标识您的机器人。                                   │
│  与OpenJiuwen身份认证无关。                                          │
│  在正常操作期间不会过期(除非您主动轮换)。                             │
└──────────────────────────────────┬──────────────────────────────────┘
                                   │
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第2步 — 启动连接器服务器  (开发者运行)                                  │
│                                                                     │
│  python -m connect.adapters.channels.run <platform> [OPTIONS]       │
│                                                                     │
│  示例:                                                              │
│    python -m connect.adapters.channels.run telegram                 │
│      --token TELEGRAM_BOT_TOKEN                                     │
│      --backend-url http://localhost:8000                            │
│                                                                     │
│    python -m connect.adapters.channels.run whatsapp                 │
│      --access-token WHATSAPP_ACCESS_TOKEN                           │
│      --phone-number-id 123456                                       │
│      --verify-token my_verify_secret                                │
│      --backend-url http://localhost:8000                            │
│      --port 8080                                                    │
│                                                                     │
│  内部发生的事情:                                                     │
│    launcher.py从参数中读取平台凭据                                   │
│    创建OpenJiuwenClient(base_url=backend_url)                       │
│      → 尚无OpenJiuwen令牌 — 用户尚未登录                            │
│    注册所有命令处理器(login、logout、agents等)                        │
│    启动服务器(见第3步)                                               │
│                                                                     │
│  进程持续运行直到您停止它(Ctrl+C或服务管理器)。                       │
│  一个运行中的进程同时服务该平台的所有用户。                           │
└──────────────────────────────────┬──────────────────────────────────┘
                                   │
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第3步 — 平台连接  (两种模式，取决于平台)                              │
│                                                                     │
│  模式A — SDK/长轮询(Telegram、Slack、Discord):                       │
│                                                                     │
│    平台SDK向平台服务器打开持久连接并通过它接收消息。                   │
│    不需要公共URL。机器人主动连接；平台推送事件回来。                   │
│                                                                     │
│    Telegram示例:                                                    │
│      python-telegram-bot Application开始轮询                        │
│        → 长轮询Telegram服务器获取更新                               │
│        → Telegram将新消息推送给机器人                               │
│                                                                     │
│  模式B — Webhook(WhatsApp、Teams、Twilio、Messenger、WeChat):        │
│                                                                     │
│    连接器服务器运行FastAPI+uvicorn HTTP服务器。                        │
│    消息平台每当用户发送消息时向您的公共URL发送POST请求。              │
│    需要公网可访问的URL(真实域名或ngrok隧道)。                        │
│                                                                     │
│    WhatsApp示例:                                                    │
│      uvicorn在8080端口启动                                          │
│        → 您在Meta开发者控制台注册                                   │
│          https://yourdomain.com/webhook                             │
│        → Meta为每条收到的消息发送POST /webhook                      │
│                                                                     │
│  两种模式的结果相同: 用户消息到达连接器机器人并被路由到正确的处理器。    │
└──────────────────────────────────┬──────────────────────────────────┘
                                   │  用户发送消息
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第4步 — 用户身份认证  (每个用户，每个设备/会话一次)                   │
│                                                                     │
│  当新用户首次给机器人发消息时，他们尚未登录。                         │
│  机器人还没有他们的OpenJiuwen令牌。                                  │
│  他们必须进行身份认证。                                              │
│                                                                     │
│  /login流程(所有平台逻辑相同):                                       │
│                                                                     │
│  4a. 用户在聊天中输入/login(或"login")。                             │
│      机器人回复: "请输入您的OpenJiuwen用户名:"                       │
│                                                                     │
│  4b. 用户输入他们的OpenJiuwen用户名。                                │
│      机器人将其临时存储在user_data中并询问密码。                      │
│                                                                     │
│  4c. 用户输入他们的OpenJiuwen密码。                                  │
│      机器人调用: do_login(backend_client, username, password)        │
│        → POST /auth/login 到OpenJiuwen后端                          │
│        → OpenJiuwen返回: access_token, refresh_token                │
│        → backend_client.set_token(access_token)                     │
│        → GET /spaces/ 找到用户的默认空间                            │
│        → 返回{token, space_id, refresh_token}                      │
│                                                                     │
│  4d. 机器人调用: set_user_data(user_id, token, space_id,            │
│                               refresh_token)                        │
│        → 保存到磁盘上的.{platform}_tokens.json，以user_id为键       │
│        例如 .telegram_bot_tokens.json:                             │
│          {                                                          │
│            "123456789": {                                           │
│              "token": "eyJhbGc...",                                 │
│              "space_id": "space-abc",                               │
│              "refresh_token": "eyJhbGc..."                          │
│            }                                                        │
│          }                                                          │
│                                                                     │
│  4e. 机器人告知用户: "✅ 已以alice身份登录。现在可以运行             │
│      智能体和工作流了。"                                             │
│                                                                     │
│  此令牌在机器人重启后仍然有效。除非令牌过期且                        │
│  自动刷新也失败(见步骤5c)，否则用户在下次会话中不需要                │
│  重新登录。                                                         │
└──────────────────────────────────┬──────────────────────────────────┘
                                   │  用户发送命令
                                   ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第5步 — 每条消息的处理流程  (用户发送的每个命令)                      │
│                                                                     │
│  示例: 用户输入"/workflow_execute wf-99"                             │
│                                                                     │
│  5a. 平台将消息推送给连接器机器人。                                    │
│      workflow_execute的处理器被调用。                                │
│                                                                     │
│  5b. 机器人从磁盘加载用户的令牌:                                     │
│        token    = get_user_token(user_id)      ← 来自.json文件      │
│        space_id = get_user_space_id(user_id)                        │
│        refresh  = get_user_refresh_token(user_id)                   │
│        backend_client.set_token(token)                              │
│        backend_client.set_space_id(space_id)                        │
│                                                                     │
│  5c. 机器人调用: verify_and_refresh(client, user_id, refresh)       │
│      令牌仍然有效 → 继续                                            │
│      令牌过期，refresh_token存在:                                    │
│        → POST /auth/refresh with refresh_token                      │
│        → OpenJiuwen返回新的access_token                             │
│        → client.set_token(new_token)                                │
│        → .json文件静默更新                                          │
│        → 用户看不到任何变化 — 一切正常工作                           │
│      令牌过期，刷新也失败:                                           │
│        → 机器人告知用户: "会话已过期。请重新/login。"                │
│        → 流程在此中止                                               │
│                                                                     │
│  5d. 机器人调用客户端逻辑:                                          │
│        execute_workflow(client, "wf-99", inputs={...})              │
│          → POST /api/v1/execution/workflow                          │
│          → Authorization: Bearer eyJhbGc... (此用户的令牌)           │
│          → OpenJiuwen以该特定用户身份执行工作流                      │
│          → 响应以SSE事件流形式返回                                   │
│        parse_workflow_result(events) → (outputs, error)             │
│                                                                     │
│  5e. 机器人为平台格式化输出并回复:                                   │
│      Telegram → 发送文本消息                                        │
│      Slack    → 发送Slack消息块                                     │
│      WhatsApp → 通过Meta Graph API发送WhatsApp消息                  │
│      CLI      → 打印到stdout                                        │
│                                                                     │
│  每个用户的令牌只用于他们自己的请求。                                │
│  用户A的令牌永远不会触及用户B的请求。                                │
└─────────────────────────────────────────────────────────────────────┘
```

```
              ┌─────────────▼────────────────────┐

                            │
用户看到:   "✅ 工作流已完成。输出: ..."
```

## 令牌生命周期 — 连接器 vs. MCP

| | 连接器 | MCP |
|---|---|---|
| 谁进行认证 | 每个人类用户，在运行时交互式进行 | 开发者，一次，在配置时 |
| 何时 | 用户首次给机器人发消息时 | 在Claude Desktop启动之前 |
| 如何 | /login → 在聊天中输入用户名+密码 | 令牌粘贴到claude_desktop_config.json |
| 存储位置 | 磁盘上的`.{platform}_tokens.json`，以user_id为键 | OS进程内存(启动时从配置读取) |
| 多用户 | 是 — 同一文件中每用户一个令牌 | 否 — 配置的唯一身份只有一个令牌 |
| 令牌过期 | 通过refresh_token自动处理(静默) | 不处理 — 需要手动更新配置并重启 |
| 登出 | /logout从.json文件中清除该用户的条目 | 无登出 — 停止Claude Desktop/从配置中移除 |

## 用户令牌过期时发生什么

```
用户发送任意命令
        ↓
机器人从.json文件加载令牌
        ↓
调用verify_and_refresh()
        ↓
  ┌─────────────────────────────────────┐
  │ 令牌仍然有效                         │ → 正常继续(最常见)
  └─────────────────────────────────────┘
  ┌─────────────────────────────────────┐
  │ 令牌过期 + refresh_token存在         │ → POST /auth/refresh
  │                                     │   新令牌保存到.json
  │                                     │   用户看不到任何变化
  └─────────────────────────────────────┘
  ┌─────────────────────────────────────┐
  │ 令牌过期 + 刷新也失败                │ → 机器人告知用户:
  │                                     │   "会话已过期。请重新/login。"
  └─────────────────────────────────────┘
```

与MCP不同，用户可以立即通过输入/login恢复 — 无需开发者干预，
无需编辑配置文件，无需重启。
